# 架构设计题

## 模块一：Context / Performance

### 1.1 长窗口 / 多模态下 First Token 加速

> 大模型面对第一轮长窗口或多模态输入时，first token 会显著变慢。有什么快速/低成本/用户体验也不差的方案？从 5-10 秒稳定压缩到 2 秒。

**思考方向：**
大模型面对第一轮长窗口或多模态输入时，first token 会显著变慢的原因是大模型处理大量信息的物理计算延迟，第一轮代表没有缓存可以用。要优化的话先考虑整个生命周期，接收输入->读取和加载->传给LLM->LLM回复->前端渲染，然后考虑优化点。接收输入层的优化应该可以做预处理，边上传变解析；读取和加载因为第一轮没有缓存，应该没有优化，如果业务场景有缓存的话可以使用缓存；传给LLM：正常来说是把整个输入传给LLM，有没有办法能够避免全量传输，要分问题讨论，可以考虑用一个小模型对用户问题做分类，对于检索这类问题可以只用传片段，对于总结这类问题要全部；LLM回复：在等待LLM回复之前可以考虑使用一套预定义回复，类似豆包，用小模型或者程序来做；前端渲染：流式输出，添加进度条等


**答案：**

核心思路是从全生命周期五个环节分别优化：

1. **输入层**：边上传边解析，流水线处理，不等全量到达。
2. **加载层**：文件哈希去重、热点文档全局缓存、图片编码缓存，虽然首轮无推理缓存，但应用层能做缓存。
3. **传输层（最大收益）**：区分问题类型——检索型走 RAG 仅传 top-k 片段，总结型才传全量。用规则匹配或小模型做意图分类。
4. **回复层**：不等 LLM 出首字，先回一句预定义确认语（如"已收到，正在分析……"），用户感知 first token 降至 50ms 内。
5. **渲染层**：骨架屏 + 流式打字，整体降低感知延迟。

多模态额外优化：图片筛选丢弃无关图、降采样、编码缓存。

检索型问题通过检索精简 + 确认语料可稳定在 1 秒内；总结型问题物理延迟仍较高，但用户感知可压缩到 2 秒以内。

### 1.2 200 轮 Session 的 Context 压缩

> 一个 session 连续聊了 200 轮，context 快爆了。你会怎么做压缩？如何确保压缩后的对话仍然流畅？

**思考方向：**

直接截断不行，会丢失前期约定和关键事实，应将旧对话提炼为结构化摘要注入 prompt。触发时机：不宜等到 context 快爆时才紧急压缩，应采用水位触发——剩余 tokens 低于阈值（如 20K）时，将最早的 N 轮压缩为摘要。但要注意摘要本身占 tokens，所以阈值和摘要大小的关系要平衡，可能需要多轮渐进压缩。
摘要格式：保留关键事实。格式为"时间 + 决策/事实/偏好"的结构化列表，如"用户偏好：不吃鱼 | 出发日期：12月10日 | 已预定：新宿酒店"。识别"关键事实"需要 Agent 在每轮对话后做一次轻量判断：本轮是否产生了需要记住的信息？
流畅性保证：压缩可能导致"刚才那个方案"这类指代断裂。解决方案：1. 保留最近 N 轮原文不做压缩；2. 压缩时要求 LLM 进行指代消解，将"那个方案"等描述展开为具体描述。

**答案：**

1. **触发策略**：水位触发，剩余 tokens 低于阈值（如 20K）时将最早 N 轮压缩。渐进式推进——首次压缩 1-150 轮，对话继续后再次触发时压缩 151-180 轮，始终保留最近 30-50 轮原文不动。
2. **摘要格式**：结构化事实列表而非话题概括。格式为"时间 + 决策/事实/偏好"，如"用户偏好：不吃鱼 | 出发日期：12月10日 | 已预定：新宿酒店"。Agent 每轮对话后做轻量判断：本轮是否产生了需要记住的事实？有则追加到摘要，减少后续全量压缩的压力。
3. **摘要放在 prompt 的什么位置**：System prompt 中插入一个"对话历史摘要"区块，位于最近原文之前。LLM 看到的结构是：system prompt → 摘要 → 最近 N 轮原文 → 当前用户消息。
4. **流畅性保证**：保留最近 30-50 轮原文不压缩；压缩时要求 LLM 做指代消解，将"刚才那个""上次说的"等模糊指代展开为具体内容。

## 模块二：Memory

### 2.1 Memory 召回策略

> 和聊天 Agent 熟悉半个月后，用户问了一个以前问过的问题。Agent 如何做 memory 召回更合理？

**思考方向：**

单纯靠语义相似度召回不够。用户不会用相同措辞重复旧问题——"推荐北京日料"和"上次那家吃鱼的"语义相似度可能很低，但指向同一件事。需要多路召回融合：语义向量捕捉意图层面匹配，关键词 BM25 捕捉字面重合，时间衰减拉高近期记忆权重，实体链接将"那家"→"日料店"做指代消解。
时间权重不能简单线性衰减。越近越相关是一般规律，但某些高重要性记忆（如"定好了婚期12月"）即使过去很久也不能被淹没。召回分数 = 语义相似度 × 时间新鲜度 × 记忆重要性，三者加权融合。
召回结果直接拼接在末尾会干扰 LLM 对当前对话的理解。更好的做法是在 system prompt 中开辟独立区块，结构化呈现："【相关历史记忆】\n1. [2024-06-20] 用户偏好日料，忌生食\n2. [2024-06-15] 预算 500 元/人"，让 LLM 自主选择。
结构化事实做索引，召回时匹配快、噪音低；原始对话做上下文补充，返回时一并提供，帮 LLM 还原完整语境。

**答案：**

1. **多路召回**：语义向量 + BM25 关键词 + 实体链接，三路并行后合并去重。语义负责意图匹配，关键词负责字面重合，实体链接负责指代消解（"那家"→"日料店"）。
2. **时间加权**：召回分数 = 语义相似度 × 时间新鲜度 × 记忆重要性。近期优先但重要性兜底——婚期、预算等关键决策即使久远也不被淹没。
3. **结构化注入**：不在对话末尾拼结果，而在 system prompt 中独立区块呈现，如"【相关历史记忆】\n1. [06-20] 日料偏好 | 2. [06-15] 预算500"。LLM 自主选择引用哪些，不强制消费。
4. **双层存储**：结构化事实建索引（检索快、噪音低），原始对话存全文（召回后附带给 LLM 补全语境）。

### 2.2 Memory 经典框架与发展趋势

> 你理解的 Agent memory 经典框架是什么？它的发展趋势是什么，最头部的玩家在怎么做？

**思考方向：**

经典框架可从计算机体系结构类比理解：工作记忆 = context window（当前可见，容量有限），短期记忆 = session 内摘要/对话历史（本次会话），长期记忆 = 向量数据库等持久存储（跨会话）。操作层面包含四个动作：写入（什么值得记）、更新（信息变化时修正）、检索（用时怎么找）、遗忘（什么可以丢）。
趋势上，"更大 context"和"更聪明检索"两条路线并行发展，不是二选一。长窗口降低了检索频率但不消灭检索需求——成本和延迟仍然是约束。另一个趋势是 memory 从"被动存储"走向"主动更新"：Agent 不仅记，还会在信息变化时主动修正旧记忆。
头部玩家各有不同的设计哲学。OpenAI ChatGPT Memory：自动从对话中提取关键信息存入 memory，用户可查看/编辑/删除，天然跨 session 生效，设计哲学是"隐式、自动化"。Anthropic Claude：通过 MCP 协议让 Agent 连接外部工具和数据库，memory 不是内置功能而是"工具连接"，设计哲学是"显式、开发者可控"。Google：押注超长 context（Gemini 2M tokens），试图让 memory 问题被足够大的窗口消解，NotebookLM 走知识库路线。

**答案：**

1. **经典框架**：类比计算机存储层次——工作记忆（context window，当前可见，容量有限）→ 短期记忆（session 摘要，本次会话）→ 长期记忆（向量库持久存储，跨会话）。四个核心操作：写入（什么值得记）、更新（信息变化时修正）、检索（多路召回）、遗忘（过期淘汰）。
2. **三家设计哲学**：OpenAI 走隐式自动化路线，自动提取记忆，用户可查看/编辑但无需主动介入；Anthropic 走显式工具化路线，通过 MCP 协议把 memory 变成外部可插拔能力，开发者完全自主控制；Google 走超长窗口路线，用 Gemini 2M tokens 让大部分 memory 需求在单次 context 内消解，配合 NotebookLM 覆盖知识库场景。
3. **趋势**：更大 context 和更智能检索并行发展，不互斥。长窗口降低检索频率但不消灭检索需求——成本和延迟仍是硬约束。此外 memory 正从"被动存储"走向"主动更新"：Agent 不等人问才去检索，而是在信息变化时主动修正旧记忆。

## 模块三：Task

### 3.1 长程任务的目标遗忘问题

> 对于长程任务，大模型执行一段时间可能会忘掉目标，你知道哪些解决方案，有什么优缺？

**思考方向：**
忘记目标是注意力偏移了，有研究表明LLM 对 context 中间位置的信息识别能力较弱，大模型执行一段时间目标可能沉到中间了，考虑每隔 N 轮就把目标重新注入到 context 尾，强制回到注意力热点区，但是如果目标太大会占用很多context，考虑分解任务，每完成一个子任务就自检一次状态，目标注入采用状态快照格式记录各个子任务的完成情况，tokens占用小；还可以加入监督agent，在任务开始时记住模板，不和主 Agent 共享 context，只读当前状态和原始目标，判断目标是否偏离，缺点是引入额外LLM；

**答案：**

根因是 LLM 的 "Lost in the Middle" 效应——对 context 中间位置的信息识别能力弱，长程执行中目标逐渐被淹没。三种方案各有取舍：

1. **滚动重注入**：每隔 N 轮将目标重新注入 context 尾部（用户消息区，非 system prompt），强制回到注意力热点。优点实现简单；缺点是完整目标反复注入挤占 context，因此必须搭配压缩，用 50 token 的状态快照代替完整目标描述。

2. **任务分解 + 检查点**：执行前将大任务拆为子任务列表，每完成一个子任务做一次自检，对照原始目标判断是否偏离。优点是把"持续记忆"转化为"阶段性核对"，降低注意力负担；缺点是拆解本身依赖 LLM 质量，子任务粒度不好控制，且每次自检多一次 LLM 调用。

3. **外部监督 Agent**：独立运行的轻量 Agent，只持有原始目标和当前状态，不与主 Agent 共享 context，判断是否偏离。优点是不受主 Agent context 淹没影响；缺点是引入额外 LLM 成本，监督者自身也可能误判，架构复杂度增加。

三种方案可叠加：先拆子任务 → 执行中用状态快照滚动注入 → 关键节点由监督 Agent 做最终审核。

### 3.2 定时复盘总结 Agent

> 用户给 Agent 下达任务：每天早上 9 点根据昨天聊天情况做复盘总结。你会怎么设计？

**思考方向：**

这个任务和即时对话不同，是定时触发的批处理流程。首先要有定时触发器，可以用系统 cron 或者 Agent 内置的调度器，每天早上 9 点按用户时区触发，要考虑如果 9 点进程刚好挂了的情况，记录上次执行日期，漏了就补跑。聊天记录存在数据库里，按用户和昨天的日期范围查询，但不能直接把全量消息扔给 LLM，token 会爆而且噪音太多，需要过滤系统消息，如果消息太多可以先做一轮预摘要。总结生成可以分两步，先对每个 session 生成简短摘要，再汇总所有 session 摘要生成最终复盘，prompt 里要指定总结维度比如关键决策、待办事项、用户情绪、未解决问题。还要考虑异常，比如昨天没聊天就生成一个空通知，LLM 调用失败就重试几次，结果推送给用户，在线就发消息，不在线就存通知。

**答案：**

1. **定时触发**：系统 cron 或内置调度器，按用户时区每天 9:00 执行。记录上次成功日期，漏执行自动补跑，保证幂等。
2. **数据提取**：从数据库按 user_id + 昨日日期范围查聊天记录。过滤系统消息和空消息，消息量超阈值时先做预摘要，避免 token 爆炸。
3. **总结生成**：两阶段——先对每个 session 生成 100 字摘要，再汇总所有 session 摘要生成最终复盘。prompt 指定维度：关键决策、待办事项、用户情绪、未解决问题。
4. **容错与交付**：昨日无聊天则生成空通知。LLM 调用失败重试 3 次。结果推送：用户在线直接发送，不在线存入待读通知。

## 模块四：Tool / Session Runtime

### 4.1 异步工具执行与通知

> Agent 工具有同步和异步两类。异步工具不能让用户一直等，但结果依然重要。你会如何设计异步工具执行和完成通知？

**思考方向：**

参考操作系统的进程调度模型。Agent 调用异步工具后立即返回确认语，任务进入阻塞队列等待执行，完成后放入就绪队列。调度采用非抢占式——因为 LLM 输出流打断后已吐出的 token 无法收回，只能在当前回复完成后才处理就绪队列中的结果。如果用户在等待期间发了新消息，不打断当前对话，继续处理新消息，等结果就绪后再以"中断"方式插入对话流。结果展示不直接拼进对话正文，而是开辟独立区域（类似 IDE 的任务中心或通知面板），Agent 基于结果生成一句自然语言摘要贴在对话里，详情让用户点开看。

**答案：**

1. **调用后立即返回**：Agent 调用异步工具后不等结果，直接回一句确认语（如"已开始生成报表，完成后通知您"），任务放入阻塞队列，释放对话流。
2. **状态追踪**：用任务 ID 追踪生命周期——pending → running → done/failed。就绪队列存放已完成待投递的结果。
3. **非抢占式调度**：结果就绪后不打断当前 LLM 输出，等本轮回复完成后才从就绪队列取出投递。因为流式输出无法安全中断。
4. **等待期间新消息**：不阻塞对话，继续处理。结果回来时以"系统通知"形式插入对话流，Agent 生成一句自然语言摘要（如"报表已生成，本月收入环比增长 12%"），详情放独立面板。
5. **失败处理**：异步任务超时或失败，Agent 主动告知用户并给出建议（重试/降级/手动处理）。

### 4.2 Session Busy 时的并发处理

> 如果 session state 为 busy，此时用户又发来新消息，或者异步工具完成事件也到达，runtime 应该如何处理？

**思考方向：**

直接用操作系统的进程状态模型映射 session 状态：idle（空闲，等待输入）、running（LLM 正在推理）、blocked（等待异步工具返回）。busy 就是 running 或 blocked 状态，此时新事件不打断，排队进入就绪队列。用户新消息和异步工具完成事件都在同一个就绪队列里，按到达时间 FIFO 排序，不区分优先级——因为区分优先级会引入调度策略的复杂度，而且大部分场景下用户消息和工具结果的重要性无法自动判断。当前 LLM 推理完成后，runtime 从就绪队列取出队首事件投递给 Agent 处理。排队期间如果用户连续发了多条消息，就绪队列里会堆积多个用户事件，Agent 依次处理。

**答案：**

1. **状态模型**：idle（空闲）→ running（LLM 推理中）→ 完成后回 idle。异步工具等待期间为 blocked，事件就绪后入队列。
2. **排队机制**：busy 状态下所有事件（用户新消息、工具完成通知）统一进入就绪队列，按到达时间 FIFO 排序，不区分优先级，避免调度策略复杂化。
3. **调度时机**：当前 LLM 推理完成后，runtime 从就绪队列取队首事件投递。连续多条用户消息按顺序依次处理，Agent 逐个回复。
4. **与 4.1 的事件流转**：异步工具调用 → 阻塞队列（4.1）→ 就绪队列（4.1）→ 与用户新消息混合进入统一就绪队列（4.2）→ FIFO 依次投递。

## 模块五：Agent Runtime 架构对比

### 5.1 Claude Code vs OpenAI Function Calling

> Claude Code 的工具输出方式和国内 GLM / 豆包等 OpenAI-compatible function calling 有什么不同？他们各自这样设计的优缺点是什么？

**思考方向：**

OpenAI 的 function calling 把文本和工具调用分成两个独立字段：content（文本）和 tool_calls（工具调用数组），一个 response 只能要么有文本要么有工具调用，finish_reason 标记为 tool_calls 时 content 为空。GLM、豆包等国内模型兼容这套协议。Claude 则把工具调用作为一种 content block 类型，和文本 block 混在同一个 content 数组里，可以先说一段话再调工具，调完再继续说，不需要像 OpenAI 那样分两轮。并行多工具调用两边都支持，但 OpenAI 是 tool_calls 数组一次性返回，Claude 是同一个 content 里的多个 tool_use block。流式输出上，OpenAI 用户先看到文本，突然结束然后等工具执行完再继续输出；Claude 用户看到的是"先解释→调工具→继续输出"的连续流，更自然。两种设计的本质差异是：OpenAI 把工具调用当作对话流的"断点"，Claude 把它当作对话流的"内嵌动作"。

**答案：**

1. **结构差异**：OpenAI 将文本（content）和工具调用（tool_calls）分离为两个独立字段，一轮响应只能出其一；Claude 将工具调用作为 content block 类型，与文本 block 混排，一轮响应可先文-后调用-再文。GLM/豆包等国内模型兼容 OpenAI 协议。
2. **OpenAI 优点**：结构清晰，客户端处理简单，并行多工具调用直观（一个 tool_calls 数组）。缺点：无法"先解释再执行"，需要两轮（第一轮 content 为空只出 tool_calls，第二轮基于结果生成文本）。
3. **Claude 优点**：对话流更自然，LLM 可以先说"让我查一下"→调工具→基于结果继续说，流式体验连贯。缺点：协议更复杂，开发者需处理混合 block 类型的解析。
4. **本质**：OpenAI 把工具调用看作"对话暂停，去执行"，Claude 把它看作"对话的一部分"。前者简单可控，后者自然连贯。

### 5.2 OpenHands 状态机设计

> OpenHands 的状态机设计有什么优缺？更优雅的实现方式是怎么样的？

**思考方向：**

OpenHands 用平面状态机管理 Agent 运行时，状态如 init → running → awaiting_user_input → paused → finished → error，状态间有明确的转换规则。优点是一目了然，所有状态和转移路径可枚举，调试时容易定位当前在哪、怎么到这里的，测试时覆盖所有状态转换即可。缺点是平面状态机只能表达"单一状态"，遇到复合场景就吃力——比如 Agent 一边 executing_code 一边 waiting_for_user_input，平面状态机必须二选一，丢失了语义。替代方案有层级状态机，允许状态嵌套，running 下面可以有 waiting_for_tool 和 executing_code 子状态；或者用 Actor Model，不用全局状态机，每个组件自己管理自己的状态靠消息通信，更自然地表达并发。但替代方案的代价是复杂度增加——层级状态机要处理状态进入/退出的副作用，Actor Model 要处理消息丢失和顺序问题。

**答案：**

1. **优点**：状态可视化、转移路径可枚举、调试和测试成本低，适合状态少且互斥的简单场景。
2. **缺点**：无法表达复合状态——当 Agent 同时"执行代码"且"等待用户确认"时，只能选其一，丢失并发语义。状态数量随着场景膨胀时会爆炸式增长。
3. **层级状态机**：允许状态嵌套，running 下拆出 waiting_for_tool、executing_code 等子状态，并发场景可分别建模。代价是进入/退出子状态时的副作用管理更复杂。
4. **Actor Model**：不用全局状态机，每个组件（工具执行器、用户交互、LLM 推理）各自维护状态，靠消息驱动。天然支持并发，但引入分布式系统的经典问题：消息顺序、丢失、重复投递。
5. **选型建议**：简单场景平面状态机足够；状态超过 10 个且出现并发现象时考虑层级状态机；组件间需要独立生命周期时考虑 Actor Model。
